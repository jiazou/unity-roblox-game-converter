# Component Analysis Comparison — 6-Component Breakdown vs Master Plan & Implementation

> Date: 2026-03-02
> Scope: Compare the "Automated Scene & Asset Bridge" 6-component breakdown against current docs and code

---

## Overview

The new analysis takes a **platform-architecture** perspective, organizing the work into 6 components that follow data flow from Unity → middleware → Roblox Cloud → Roblox Studio. This differs from the master plan's **module-per-concern** structure (11 Python modules orchestrated by `converter.py`). The differences are significant and reveal both gaps and deliberate architectural choices.

---

## Component-by-Component Comparison

### 1. Unity Editor Extractor Plugin (C#)

The new analysis calls for a **native C# plugin running inside the Unity Editor** with four sub-features.

| Sub-feature | New Analysis | Master Plan / Implementation | Difference |
|---|---|---|---|
| **Scene Traversal** | C# script inside Unity to scan the Scene Graph | `scene_parser.py` parses `.unity` YAML files externally | **Different approach** |
| **Spatial Data Serialization** | Extracts Transform matrices from Unity runtime | `scene_parser.py` extracts `m_LocalPosition`, `m_LocalRotation`, `m_LocalScale` from YAML | **Different approach** |
| **Asset Exporter** | Auto-exports meshes as .fbx/.glb from Unity | `asset_extractor.py` discovers existing files in `Assets/` | **Different approach** |
| **UI Parsing** | Captures Canvas/RectTransform Anchor data | `scene_parser.py` handles RectTransform (classID 224) but **no UI-specific extraction** | **Gap in implementation** |

**Why the difference exists:**

The current tool is designed as an **external, offline pipeline** — it reads Unity project files directly from disk without requiring Unity Editor to be running. This is a deliberate architectural choice with tradeoffs:

| | External Pipeline (current) | Unity Editor Plugin (proposed) |
|---|---|---|
| **Requires Unity installed** | No | Yes |
| **Handles binary assets (.fbx)** | Reads existing files | Can export from Unity's internal format |
| **Scene data accuracy** | Parses raw YAML (no runtime evaluation) | Full runtime object graph |
| **Computed properties** | Cannot evaluate (e.g., Animator state) | Full access |
| **Deployment** | `pip install` and run | Requires Unity project to be openable |
| **CI/CD friendly** | Yes (headless) | Requires Unity batch mode |

**Assessment:** The external approach is correct for the POC use case. Unity projects store all scene data as YAML and all assets as files — the external parser captures the same information that a C# plugin would, without requiring Unity to be installed. The only case where a Unity plugin wins is **computed properties** (runtime-evaluated animations, procedural generation) which are rare in static scene conversion.

**Proposed change:** None for the core pipeline. However, the UI Parsing gap is real — see Component 4.

### 2. Asset Optimization & Processing Middleware

| Sub-feature | New Analysis | Master Plan / Implementation | Difference |
|---|---|---|---|
| **Mesh Decimation** | Algorithmic or AI decimation to ≤20K triangles | `mesh_decimator.py` — quadric decimation to ≤10K faces via trimesh | **Different limits** |
| **Texture Atlasing & Downscaling** | Combine textures to power-of-2, PBR suffixes | `material_mapper.py` processes textures individually, no atlasing | **Gap: no atlasing** |

**Why the differences exist:**

1. **Triangle limit discrepancy (20K vs 10K):** The new analysis states 20,000 triangles; our implementation uses 10,000 faces. **Both are wrong in different ways.** The actual Roblox MeshPart limit is 10,000 **polygons** (which is approximately 20,000 triangles for tri-only meshes, since 1 quad = 2 tris). The implementation is conservative (10K faces = safe for any face type). The new analysis conflates triangles with the polygon limit. The current implementation's 10K face target is the safer choice.

2. **Texture atlasing:** The new analysis calls for atlas packing with PBR suffixes (`_ALB`, `_NOR`, `_RGH`, `_MET`). The implementation processes textures per-material, which aligns with how Roblox `SurfaceAppearance` works (one material per MeshPart, each with its own texture set). Atlasing would only help if multiple materials are baked into a single MeshPart, which requires UV remapping — a complex operation that's rarely needed.

3. **PBR suffix convention:** The new analysis mentions `_ALB`, `_NOR`, `_RGH`, `_MET` suffixes. The implementation uses descriptive output filenames generated by `material_mapper.py` (e.g., `MaterialName_colormap.png`, `MaterialName_roughness.png`). This is a naming convention difference, not a functional one.

4. **Texture size limits:** The new analysis says 512×512 or 1024×1024 (2048 for avatars). The implementation allows up to 4096×4096 (`config.TEXTURE_MAX_RESOLUTION`). **As of December 2025, Roblox updated the texture resolution limit to 4096×4096** (per devforum announcement). The implementation's 4096 value is correct. The new analysis's lower limits are outdated.

**Proposed changes:**
- ~~**Fix `config.TEXTURE_MAX_RESOLUTION`**~~ — 4096 is correct as of Dec 2025
- **No atlasing needed** — per-MeshPart materials don't benefit from it

### 3. Open Cloud API Uploader

| Sub-feature | New Analysis | Master Plan / Implementation | Difference |
|---|---|---|---|
| **Authentication** | API key with `asset:write`/`asset:read` scopes | `roblox_uploader.py` uses `x-api-key` header | **Same approach** |
| **Rate Limit Manager** | Monitors `x-ratelimit-remaining`/`reset` headers, exponential backoff | `modules/retry.py` provides backoff but **no header monitoring** | **Gap: no header-based rate limiting** |
| **ID Mapping** | Captures returned `assetId`, maps back to Unity asset | `UploadResult.asset_ids` dict maps `local_name → rbx_asset_id` | **Implemented** |
| **20 MB payload limit** | Mentioned as hard constraint | Not checked in implementation | **Gap: no size validation** |

**Why the differences exist:**

The `retry.py` module was designed as a generic transient-failure retry mechanism. It retries on `ConnectionError`/`TimeoutError`/`OSError` but doesn't parse HTTP response headers for rate limit information. The Roblox API returns specific headers (`x-ratelimit-remaining`, `x-ratelimit-reset`) that allow proactive throttling rather than reactive retry-on-429.

**Proposed changes:**
- **Add rate limit header parsing** to `roblox_uploader.py` — read `x-ratelimit-remaining` and `x-ratelimit-reset` from responses, sleep proactively when approaching limits
- **Add payload size validation** — check file size before upload, warn/skip if >20 MB
- **Add HTTP 429 to retryable exceptions** in the retry configuration

### 4. Translation & Math Engine

| Sub-feature | New Analysis | Master Plan / Implementation | Difference |
|---|---|---|---|
| **UI Translator** | RectTransform anchors → UDim2 (Scale + Offset) | **Not implemented** | **Gap** |
| **Physics Mapper** | BoxCollider/SphereCollider → BasePart + WeldConstraints | **Not implemented** | **Gap** |

**Why the differences exist:**

The current implementation focused on the **visual pipeline** (meshes, materials, textures) and **code pipeline** (C# → Luau). Unity's UI system (Canvas/RectTransform) and physics colliders were lower priority for the POC, especially since:

1. **UI:** Many Unity games use third-party UI frameworks (TextMesh Pro, DOTween) that don't map cleanly to Roblox ScreenGui. The scene_parser already recognizes RectTransform (classID 224) but doesn't extract UI-specific anchor data.

2. **Physics colliders:** Unity BoxCollider/SphereCollider data is available in the scene YAML (classIDs: BoxCollider=65, SphereCollider=135, CapsuleCollider=136, MeshCollider=64). The scene_parser would capture these as ComponentData, but no conversion logic translates them to Roblox BasePart collision shapes or WeldConstraints.

**Proposed changes:**
- **UI Translator:** Add a `modules/ui_translator.py` that extracts RectTransform anchor/pivot/offset data from scene_parser's ComponentData and generates Roblox ScreenGui + Frame + UDim2 Luau code. This is a well-defined mapping:
  - `anchorMin`/`anchorMax` → UDim2 Scale component
  - `offsetMin`/`offsetMax` → UDim2 Offset component
  - `pivot` → AnchorPoint
- **Physics Mapper:** Add collider-to-BasePart logic in the scene-to-part conversion. BoxCollider → Part with matching Size, SphereCollider → Part with `Shape=Ball`. WeldConstraints for compound colliders. This is moderate complexity.

### 5. Roblox Scene Reconstructor

| Sub-feature | New Analysis | Master Plan / Implementation | Difference |
|---|---|---|---|
| **File Generator** (.rbxm/.rbxmx) | Generate importable model XML | `rbxl_writer.py` generates `.rbxl` (full place XML) | **Different format, same approach** |
| **Luau Setup Script** | ModuleScript + ServerScript using InsertService | Not implemented (scripts go into place directly) | **Not implemented** |

**Why the differences exist:**

The new analysis proposes **two output modes**: a drag-and-drop model file (`.rbxm`) and a programmatic setup script that uses `InsertService` to fetch uploaded assets by ID. The current implementation takes a third approach: a complete `.rbxl` place file that can be opened directly in Roblox Studio.

| Output Format | Current (.rbxl) | Proposed .rbxm | Proposed Luau Script |
|---|---|---|---|
| Contains full scene | Yes | Yes (as model) | No (fetches at runtime) |
| Requires upload first | No | No | Yes (assets must be in cloud) |
| Can merge with existing place | No (replaces) | Yes (drag-drop) | Yes (runs alongside) |
| Offline usable | Yes | Yes | No |

**Assessment:** The `.rbxl` approach is the most self-contained and appropriate for POC. The `.rbxm` model format would be useful for merging converted content into existing Roblox places — this is a reasonable post-POC feature. The Luau InsertService approach is specifically valuable when assets have already been uploaded and you want runtime assembly.

**Proposed changes:**
- **Add `.rbxm`/`.rbxmx` output option** to `rbxl_writer.py` — the XML format is nearly identical to `.rbxl` but wraps content in a Model container instead of DataModel. Small modification (~30 lines).
- **No InsertService script needed for POC** — the direct file approach is simpler and more reliable.

### 6. Code Translation Stubber

| Sub-feature | New Analysis | Master Plan / Implementation | Difference |
|---|---|---|---|
| **Syntax Transpiler** | Source-to-source compiler or LLM | `code_transpiler.py` — tree-sitter AST + regex rules + Claude AI | **Implemented** |
| **Architecture Mapping** | Split MonoBehaviour → LocalScript + ServerScript + ModuleScript | **Not implemented** — all scripts become `Script` type | **Gap** |
| **Lifecycle Refactoring** | `Update()` → `RunService.Heartbeat` etc. | `api_mappings.py` LIFECYCLE_MAP + regex rules | **Implemented** |

**Why the differences exist:**

The **architecture mapping** gap is the most significant. Unity MonoBehaviours run in a single context (the Unity runtime), while Roblox requires explicit client/server separation:

- **LocalScript**: Runs on client (input handling, camera, UI)
- **Script** (ServerScript): Runs on server (game logic, data, physics authority)
- **ModuleScript**: Shared code imported by either side

The current implementation places ALL transpiled scripts as `Script` (server-side) in `ServerScriptService`. This means:
- Client-side code (Input handling, Camera manipulation, UI updates) will silently fail because server Scripts can't access client services
- No `ReplicatedStorage` ModuleScripts for shared logic
- No RemoteEvent bridging between client and server

**Proposed changes:**
- **Add script classification heuristic** to `code_transpiler.py`: analyze Unity API usage to determine script placement:
  - Uses `Input.*`, `Camera.*`, `Screen.*`, `GUI.*` → **LocalScript** (in StarterPlayerScripts)
  - Uses `[Command]`, `[SyncVar]`, game state mutation → **Script** (in ServerScriptService)
  - Pure utility/math functions, data classes → **ModuleScript** (in ReplicatedStorage)
- **Update `rbxl_writer.py`** to support placing scripts in `StarterPlayerScripts` and `ReplicatedStorage` (currently only `Workspace` parts and `ServerScriptService`)
- The tree-sitter AST analysis (`CSharpClassInfo.unity_apis_used`) already detects which APIs are used — this data can drive the classification

## Summary of All Differences

### Architectural Philosophy Differences (not bugs — deliberate choices)

| Aspect | New Analysis | Current Implementation | Verdict |
|---|---|---|---|
| Unity data source | C# plugin inside Unity Editor | External Python parser reading YAML files | **Current is correct for POC** — no Unity dependency required |
| Mesh tri limit | 20,000 triangles | 10,000 faces (conservative) | **Current is safer** — 10K faces covers all polygon types |
| Output format | .rbxm model file | .rbxl place file | **Current is more complete** — add .rbxm as option |
| Runtime assembly | InsertService Luau script | Direct file embedding | **Current is simpler** — defer InsertService approach |
| Texture strategy | Atlas + power-of-2 + PBR suffixes | Per-material individual textures | **Current is correct** for Roblox SurfaceAppearance model |

### Real Gaps — Status

| Gap | Severity | Component | Status |
|---|---|---|---|
| **Script client/server classification** | High | Code Translation (#6) | **FIXED** — `_classify_script_type()` in code_transpiler.py, rbxl_writer places scripts in correct containers |
| **UI Translator** (RectTransform → UDim2) | Medium | Translation Engine (#4) | **FIXED** — new `modules/ui_translator.py`, integrated into converter pipeline |
| **Rate limit header parsing** in uploader | Medium | Cloud API (#3) | **FIXED** — `_check_rate_limit_headers()` in roblox_uploader.py |
| **Payload size validation** (20 MB assets / 100 MB places) | Low | Cloud API (#3) | **FIXED** — pre-upload size checks in roblox_uploader.py |
| **Physics collider mapping** | Medium | Translation Engine (#4) | **FIXED** — BoxCollider/SphereCollider/CapsuleCollider/Rigidbody handled in converter.py |
| **Texture max resolution** | Low | Asset Middleware (#2) | **No change needed** — 4096 is correct as of Dec 2025 |
| **`.rbxm` output option** | Low | Scene Reconstructor (#5) | Deferred — not needed for POC |

### Items in New Analysis That Are Not Needed

| Item | Reason |
|---|---|
| **Unity Editor C# Plugin** | External YAML parsing achieves the same result without requiring Unity to be installed. The plugin approach adds a deployment dependency for no functional gain on static scenes. |
| **Texture Atlasing** | Roblox's material model is per-MeshPart. Atlas packing would require UV remapping on meshes, adding complexity without benefit. |
| **InsertService Luau script** | Only needed if assets are pre-uploaded and you want runtime assembly. The direct .rbxl approach is self-contained. |
| **AI mesh decimation (Tripo/Meshy)** | Quadric decimation via trimesh produces excellent results algorithmically. External AI APIs add latency, cost, and a network dependency for a solved problem. |
